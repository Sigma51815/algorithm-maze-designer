## 0.1 内置统计工具

### MazeStatistics 统计指标（`maze.cpp:504-554`）

```cpp
struct MazeStatistics {
    int solutionLength  = 0;  // 起点到终点的最短路径步数（树的直径长度）
    int deadEnds        = 0;  // 死胡同数量（度数为1的格子，不含起点和终点）
    int junctions       = 0;  // 交叉口数量（度数≥3的格子）
    int longestCorridor = 0;  // 最长无分支走廊（连续度数为2的格子链的最大长度）
};
```

这四个指标的**计算方法**：

- **死胡同**：BFS 遍历所有格子，统计 `degree == 1` 且不是起点/终点的格子数
- **交叉口**：BFS 时间统计所有 `degree >= 3` 的格子数
- **解路径长度**：从终点沿 BFS 父指针回溯到起点，统计步数
- **最长走廊**：对每个度数不是 2 的格子，沿其度数为 2 的邻居方向一直走下去直到遇到非 2 度数的格子，记录路径长度

## 1. 分治法 —— 复杂度与风格分析

### 1.1 时间复杂度

```
T(V) = 2 × T(V/2) + O(1)           均衡情况
T(V) = T(V-1) + T(1) + O(1)       退化情况（每次只切出1列/1行）
```

**详细推导：**

每个递归层只做常数工作：一次随机数生成 + 一次 `carve()` 调用。递归树结构分析：

| 情况 | 递归方程 | 解 |
|---|---|---|
| 均衡（正方形区域，交替水平/垂直切） | $T(n) = 2 \cdot T(n/2) + O(1)$ | $O(n) = O(V)$ |
| 偏斜（始终沿同一方向切） | $T(n) = T(n-1) + T(1) + O(1)$ | $O(n) = O(V)$ |

> **注**：$V = H \times W$ 表示单元格总数。无论均衡还是偏斜，总递归次数恒为 $V-1$（每次递归非基情况产生一次分割和一个通道），所以时间复杂度恒为 $O(V)$。

**常数因子**：每次递归做 2 次随机分布构造、若干次整数运算。非常小，是最快的算法。

### 1.2 空间复杂度

- **最优**（均衡分割）：递归最大深度 $\approx \log_2(\max(H, W))$，栈空间 $O(\log V)$
- **最坏**（始终沿同一方向切）：递归深度 $O(H)$ 或 $O(W)$，栈空间 $O(\sqrt{V})$（由于矩形选择的是较宽的方向切，所以最坏情况出现在 $H \gg W$ 或 $W \gg H$ 的极端矩形网格）

另外 `passages_` 数组占 $O(V)$ 空间是所有算法共有的。

### 1.3 风格特征

**宏观结构：矩形分区**

分治法生成的迷宫有最明显的**宏观结构**。从高空俯视，可以看到迷宫被层层分割为大块矩形区域：

```
┌─────────┬──────┐
│ 区域 A   │      │
│         │  C   │
├──── ─ ──┤      │
│    B    │      │
└─────────┴──────┘
```

- 区域之间仅有**唯一通道**连接（每次分割只开一个口）
- 通道的位置是随机的，但有"**瓶颈**"特征——要从 A 区去 C 区，必须经过某个特定通道
- 这导致了**探索的局部性**：玩家容易走完一个区域后才发现去下一个区域的唯一通道

**微观特征：**

- **死胡同分布**：均匀，没有明显集中区域
- **走廊长度**：中等偏短。因为分治法在每个递归层都产生连接口，而不是长走廊
- **交叉口数量**：较少。大多数格子的度数为 2（墙上通道），只有分割线上的通道口度数 ≥ 3
- **解路径**：中等长度。因为区域之间有明确的最短路径约束

### 1.4 内在特性

**数学保证：** 分治法具有**归纳正确性**。Base case 是单格，显然是一棵树（平凡生成树）。归纳步骤：两个树的并加上一条连接边 → 仍为一棵树（V 增加了，E 增加了 1，无环，连通）。

**随机性来源：**
1. 分割线的随机选择位置
2. 连接口的随机选择位置
3. 优先选宽的方向分割保证了区域尽量正方形

**生成树形态偏好：**

分治法倾向于生成"**类二分树**"的生成树。树的形态反映了分割的层次结构：
- 分割线 → 树的"瓶颈边"
- 子区域 → 子树

由于均衡分割策略，大多数区域接近正方形，连接口在区域之间的"墙"上——这导致通常是一个相对扁平的树（树的直径与区域数量相关而非单个格子数）。

## 2. Kruskal MST —— 复杂度与风格分析

### 2.1 时间复杂度

Kruskal 算法的时间由两部分组成：

```
生成候选边数组：  O(E) ≈ O(2V)
打乱候选边：       O(E) ≈ O(2V)
处理每条边：       O(E × α(V)) ≈ O(2V × α(V))
────────────────────────────────────────
总计：             O(E × α(V)) ≈ O(V)
```

**为什么 `α(V)` 可视为常数：**

$\alpha(V)$ 是阿克曼函数的反函数。实际值：
- $V = 10^3$：$\alpha(V) \approx 3$
- $V = 10^6$：$\alpha(V) \approx 4$
- $V = 10^{80}$（宇宙中原子数）：$\alpha(V) \approx 5$

对于 15×15 的迷宫（$V=225$），$\alpha(225) = 2$。

**与标准 Kruskal 的比较：**

标准 Kruskal 需要先对边按权重排序，复杂度 $O(E \log E)$。本项目通过**随机打乱**（$O(E)$）替代了排序（$O(E \log E)$），因为随机打乱等价于赋予每条边一个独立同分布的随机权重——不需要真正排序。

> 注意：UI 中的复杂度说明（`mainwindow.cpp:398`）写的是 $O(E \alpha(V))$，括号内特意注明："若采用标准 Kruskal 排序边权，则时间为 $O(E \log E)$"——这准确反映了实现细节。

### 2.2 空间复杂度

需要存储：
- 候选边数组：$O(E) \approx O(2V)$
- 并查集（parent + rank）：$2 \times O(V)$
- passages_：$4 \times V$

总计 $O(V+E)$。

### 2.3 风格特征

**宏观结构：均匀随机**

Kruskal 生成的迷宫**没有任何可识别的宏观结构**。这是它的最大特点——也是它的最大优点和最大缺点：

- **优点**：完全公平，所有可能的生成树出现的概率相等
- **缺点**：没有"分区"或"主题"结构，有时会显得"无聊"

**微观特征：**

- **死胡同分布**：均匀分布在迷宫各处，数量中等
- **走廊长度**：偏短。因为每条边被独立随机接受，长走廊的形成需要一连串"幸运"的边选择
- **交叉口数量**：中等偏多。因为随机边选择没有方向偏好
- **解路径**：中等偏长。没有人工路径缩短的结构

### 2.4 内在特性

**数学保证：均匀分布**

这是 Kruskal 最核心的数学特性：在网格图的所有可能生成树中，Kruskal 随机打乱边的方法产生的生成树**均匀分布**（uniform distribution）。证明可以在任意图论教材中找到——关键点是每条边被选入生成树的概率与其权重无关（因为打乱消除了任何顺序偏差），因此所有生成树等概率。

**这个特性意味着什么？**

如果我们从网格图的所有生成树中**完全随机地**抽一棵出来，它大概就是 Kruskal 生成的样子。这是四种算法中唯一具有这种"绝对随机"特性的算法。

**与其他三种算法的对比：**

| 算法 | 生成树分布 |
|---|---|
| 分治 | 偏向于"二分树"结构的生成树 |
| Kruskal | 均匀分布——无任何偏好 |
| DFS | 显著偏向于"长链"结构的生成树 |
| BFS 分支限界 | 偏向于"宽展开 + 长路径"的生成树，但受评分函数影响 |

### 2.5 并查集的实现质量

项目的并查集实现是**教科书级别的**——同时使用路径压缩和按秩合并：

```cpp
int find(int value) {
    if (parent_[value] != value) {
        parent_[value] = find(parent_[value]);  // ★ 路径压缩
    }
    return parent_[value];
}

bool unite(int first, int second) {
    first = find(first); second = find(second);
    if (first == second) return false;
    if (rank_[first] < rank_[second]) std::swap(first, second);  // ★ 按秩合并
    parent_[second] = first;
    if (rank_[first] == rank_[second]) ++rank_[first];
    return true;
}
```

这使得均摊复杂度严格为 $O(\alpha(V))$，接近常数时间。

## 3. 回溯 DFS —— 复杂度与风格分析

### 3.1 时间复杂度

```
初始化 visited 数组：      O(V)
主循环：
  · 每个格子入栈一次：      O(V)
  · 每个格子考虑4个邻居：    O(4V) = O(V)
  · 每次打乱邻居列表：      O(4) × V = O(V)  （邻居数 ≤ 4）
────────────────────────────────
总计：                     O(V+E) ≈ O(V)
```

详细分析：
- `visited` 保证每个格子**恰好被访问一次**（入栈即标记）
- 每个格子访问时扫描 4 个方向（总共 $4V$ 次扫描），检查 `!visited[next]`
- 回溯时从栈中移除，但不会重新考虑已访问的邻居
- 每次挑选邻居时打乱 —— neighbour 数组最多 4 个元素，打乱代价为常量

### 3.2 空间复杂度

```
visited 数组：      V 布尔值
栈：                最坏深度 V（走到底才回溯）
passages_：         4V
────────────────────────────────
总计：              O(V)
```

**栈深度最坏情况：** 如果随机选择恰好形成一条遍历所有格子的哈密顿路径，栈深度可达 V。在较小的 15×15 迷宫中（V=225），这完全不是问题；但如果在更大的网格上这可能成为限制因素。

### 3.3 风格特征

**宏观结构：一条蜿蜒的主干 + 稀疏分支**

DFS 生成的迷宫在四种算法中**最具辨识度**。其典型结构是：

```
S ──→ ──→ ──→ ──→ ──→ ──→ ──→ E
              │
              ↓
            死胡同
```

- 从起点出发，DFS 会"一条路走到黑"，形成一段很长的**主干走廊**
- 当走到死胡同时回溯，在回溯途中发现"岔路口"就分出**短分支**
- 分支再继续延伸，直到再次走入死胡同

**微观特征：**

- **死胡同数量**：**最多的**。因为每次回溯都对应一个死胡同的终点
- **走廊长度**：**最长的**。DFS 的深度优先天性保证了连续不分支的路径最大化
- **交叉口数量**：较少。大多数格子只有 2 个邻居（走廊中），只有回溯点才可能出现 3 个或更多邻居
- **解路径**：较长。主干走廊可能本身就是迷宫的直径

### 3.4 为什么 DFS 产生长走廊？

这是一个重要的直观理解点。考虑 DFS 的行为：

```
当前格子  未访问邻居数    行动
    A         3        随机挑选一个，前进
    B         2        随机挑选一个，前进
    C         0        ← 死胡同！回溯到 B
    B         1        随机挑选另一个，前进
    D         0        ← 死胡同！回溯到 B
    B         0        ← 死胡同！回溯到 A
    ...
```

关键在于：**只要还有未访问邻居，DFS 就继续前进而不是回头**。只有真的"走到头了"才回溯。这自然导致：
- 每次前进都延伸已有走廊（度数从 1→2 或 2→3）
- 回溯后如果有新方向，形成新的走廊分支
- 每个回溯点最多产生一个分支（因为回溯后就"记起了"之前剩下的邻居）

这正是 DFS 生成树天然偏向于"**深窄型**"（deep and narrow tree topology）的数学原因。

### 3.5 与递归 DFS 的对比

项目使用的是**迭代式 DFS**（显式栈），而非递归 DFS。等价性：

| 递归 DFS | 迭代 DFS（本项目） |
|---|---|
| `dfs(cell, visited)` | `stack.push(cell)` |
| 函数调用栈 | 显式 `QVector<int>` 栈 |
| 递归返回 = 回溯 | `stack.removeLast()` = 回溯 |

迭代实现避免了递归栈溢出风险，且更易于调试和可视化（`generationSteps_` 记录了每一步）。

## 4. BFS 分支限界 —— 复杂度与风格分析

### 4.1 时间复杂度

```
外层循环：          ≤ 12 次
每次尝试：
  · 重置数据：        O(V)
  · 优先队列操作：    每条边最多入队出队各 2 次（原始 + 重新入队）
                   O(E log E)
  · 评分计算：        O(V)  [statistics() 和 chooseDiameterEndpoints()]
────────────────────────────────
总计：                O(12 × E log E) = O(E log E)
```

**优先队列操作的细节：**

每条候选边最多经历：
1. **原始入队**：`addBranches()` 中将边推入优先队列（$O(\log E)$）
2. **出队检查**：`frontier.pop()` 弹出（$O(\log E)$）
3. **重新入队**（仅在 staleness 发生时）：重新计算下界后再次推入（$O(\log E)$）

Staleness 发生频率取决于迷宫的局部拥挤程度——在度数较高的交叉口附近更频繁。平均而言，每条边的处理次数在 1~2 次之间，所以总优先队列操作次数为 $O(E \log E)$。

### 4.2 空间复杂度

```
visited 数组：             O(V)
degree 数组：              O(V)
优先队列：                 最多 O(E) 个元素
passages_ + bestPassages： O(V)
────────────────────────────────
总计：                     O(V+E)
```

队列在最坏情况下可能容纳所有尚未处理的候选边。对于 15×15 迷宫，E≈420，队列最多容纳约 400 个元素——非常轻量。

### 4.3 风格特征

**宏观结构：从根节点向外"层状"扩散**

BFS 分支限界保留了 BFS 的核心特征——层次扩展。生成的迷宫从随机根节点开始向外逐层扩散：

```
        [root]
       /  |  \
     [1] [2] [3]     ← depth=1
    / |   |   | \
  ... ... ... ...    ← depth=2
  ...
```

但与纯队列式 BFS 不同，由于优先队列的下界惩罚和随机权重：

- **同层之间有随机分支选择**，不是严格按先进先出顺序
- **度数惩罚**（`degree[from] × 100`）有效防止了中心节点度数暴增
- 最终结构是"层状"但不像纯 BFS 那样"过于规则"

**微观特征：**

- **死胡同数量**：较多——评分函数 `solutionLength × 10 + deadEnds` 直接奖励死胡同
- **走廊长度**：中等偏长——评分函数间接奖励长路径
- **交叉口数量**：中等——每个格子的度数被度数惩罚间接控制在 ≤ 4
- **解路径**：**最长的**——通过 12 次重启和最佳候选保留，这是明确追求优化的目标

### 4.4 下界函数的深入分析

```
lowerBound = depth × 10000 + degree[from] × 100 + randomWeight (0~99)
```

这个公式是理解 BFS 分支限界核心特性的关键。逐项分析：

**第一项 `depth × 10000`：强制 BFS 层次纪律**

乘以 10000 保证了：任何 depth=k 的分支，其下界至少为 $k \times 10000$；任何 depth=k-1 的分支，其下界至多为 $(k-1) \times 10000 + 3 \times 100 + 99 = (k-1)\times 10000 + 399 < k \times 10000$。

因此**较低深度的分支总是先出队**——这就是 BFS 的核心保证：所有 depth=k 的格子都在 depth=k+1 之前被处理。

**第二项 `degree[from] × 100`：控制中心度数**

`degree[from]` 是从当前格子出发的现存通道数。乘以 100 在同层内产生影响：度数越低的源格子越优先。这防止了"中心节点效应"——一个格子连接了太多邻居，后来的分支会因度数高而排在后面。

**第三项 `randomWeight`：同层同度数打散**

范围 [0, 99]。仅在同层同度数的兄弟分支之间起作用（因为 `degree × 100` 范围内有 100 个值，randomWeight 在 [0,99] 只引入局部变异）。

### 4.5 Staleness 检测机制

```cpp
if (branch.sourceDegree != degree[branch.from]) {
    // 源格子的度数自该分支入队后发生了变化
    // 需要重新计算下界
    branch.sourceDegree = degree[branch.from];
    branch.lowerBound = branch.depth * 10000 + branch.sourceDegree * 100
                        + branch.randomWeight;
    frontier.push(branch);  // 重新入队
    continue;               // 跳过当前（已过期的）分支出队
}
```

这个机制保证了：每个分支在出队时，其下界是**基于当前 state 的最新估计**。它防止了以下情况：

```
时刻 T0：格子 A 度数为 0，分支 (A→B) 入队，下界 = depth×10000 + 0×100 + random
时刻 T1：分支 (A→C) 先被处理（因为 randomWeight 更小），A 度数变为 1
时刻 T2：分支 (A→B) 出队 ← 此时 A 的度数已经是 1 了！
        如果不用 staleness 检测，下界仍是基于 degree=0 的值
        这会导致不公平的优先级
```

这是典型的"**分支限界**"思想：用最新状态重新评估每个分支的价值（"限界"），再决定是否接受。

### 4.6 12 次随机重启的策略意义

为什么是 12 次？代码中没有显式说明原因，但从算法设计角度分析：

- **探索 vs 利用**：12 次独立不同起点的随机重启，增大了搜索空间覆盖率，提高了找到"好"迷宫的概率
- **最佳候选保留**：不是简单地取最后一次结果，而是取评分最高的——这是"精英保留"策略
- **提前终止**：如果某次尝试已经足够好（解路径 ≥ directDistance + 6），不再浪费计算资源
- **12 的具体数字**：可能是经验调参的结果——太少风险未探索到好的候选，太多边际收益递减

## 5. 综合对比分析

### 5.1 复杂度总览

| 算法 | 时间（表达式） | 时间（V=225, E≈420） | 空间 | 主导操作 |
|---|---|---|---|---|
| 分治 | $O(V)$ | ~225 次递归 | $O(\log V)$~$O(\sqrt{V})$ | 递归调用 |
| Kruskal | $O(E \cdot \alpha(V))$ | ~420×2 = 840 ops | $O(V+E)$ | 并查集合并 |
| DFS | $O(V+E)$ | ~225+420 = 645 ops | $O(V)$ | 栈操作 + 随机打乱4元素 |
| BFS 分支限界 | $O(E \log E)$ | ~420×9×12 ≈ 45K ops | $O(E)$ | 优先队列插入/删除 |

> 注：E log E 中 log₂(420)≈8.7。

对于 15×15 迷宫，所有算法都在毫秒级完成。差异主要在大规模网格（如 100×100 = 10,000 格）时体现。

### 5.2 风格特征总览

| 特征 | 分治 | Kruskal | DFS | BFS 分支限界 |
|---|---|---|---|---|
| 宏观结构 | 矩形分区 | 无规则 | 蜿蜒长主干 | 层状扩散 |
| 死胡同数量 | 中等 | 中等偏多 | **最多** ★ | 较多（被评分优化） |
| 走廊长度 | 中等偏短 | 偏短 | **最长** ★ | 中等偏长 |
| 交叉口数量 | 少 | 中等偏多 | 少 | 中等 |
| 解路径长度 | 中等 | 中等偏长 | 较长 | **最长** ★ |
| 可预测性 | 高（区域规律） | 低（完全随机） | 中（有主干） | 中（层次可见） |
| 自然感 | 低（太工整） | 中（随机但散乱） | **高**（像自然迷宫） | 中高（兼顾探索与挑战） |

### 5.3 生成树形态对比

将迷宫看作网格图上的生成树，各算法产生的树的**形态差异**如下：

| 算法 | 树的形态 | 解释 |
|---|---|---|
| 分治 | **平衡二分树** | 每次分割将格子群对半分开，树的高度 ≈ log V |
| Kruskal | **随机树** | 没有任何形态偏好，数学上均匀分布 |
| DFS | **深链式树** | 长而窄，树的高度接近 V（最极端可达 V-1） |
| BFS 分支限界 | **宽展式树** | 从根节点向外短距离辐射，树的高度 ≈ 直径（中等） |

### 5.4 数学原因：为什么不同算法产生不同形态？

**分治 → 平衡二分树**：每次分割是对区域的无偏划分，连接口只有 1 个，逼迫生成的树是"区域间 1 通道"结构。

**Kruskal → 随机树**：每条边被独立且等概率地考虑，没有空间或结构偏好。

**DFS → 深链式树**：深度优先搜索在网格图上的行为等同于"每次选择未访问邻居后继续走"，直到不能再走为止——这最大化了连续路径的长度。

**BFS 分支限界 → 宽展式树**：优先队列的下界按层次升序排列（`depth × 10000`），强制低深度先处理——这是 BFS 的本质特征；但度数惩罚和随机权重防止了过度集中。

### 5.5 实际应用建议

| 使用场景 | 推荐算法 | 理由 |
|---|---|---|
| 快速原型验证 | 分治 | 最快，结构清晰易调试 |
| 教学演示 | 分治或 DFS | 结构在视觉上有明确特征，方便讲解 |
| 追求公平随机 | Kruskal | 数学上的均匀分布保证 |
| 追求探索乐趣 | DFS | 长走廊 + 多死胡同，最像"真实迷宫" |
| 追求最大挑战 | BFS 分支限界 | 最长解路径，最多死路（经评分优化） |
| 作为 GA 初始种群 | 混合四种 | 增加初始多样性，让 GA 有丰富基因库 |

## 6. 可视化与动画

项目支持两种观看方式：

1. **完整展示**：一次性显示完整迷宫
2. **动画播放**：按 `generationSteps_` 记录的顺序逐步显示开墙过程（`MazeWidget` + `generationTimer_`）

动画速度为可调节的 50ms~500ms。这有助于直观感受每种算法的生成过程：
- 分治：先看到分割线上的墙被打开，然后子区域中的墙被打开
- Kruskal：随机地在各处开墙，迷宫渐渐成形
- DFS：从起点开始，开墙的区域"蔓延"出去，像一条蛇在探索
- BFS 分支限界：从某一中心点向外辐射展开

## 7. 补充说明：复杂度术语对应

在 UI 算法下拉框下方的复杂度描述中（`mainwindow.cpp:396-400`）：

| UI 算法名称 | UI 显示的复杂度 | 对应算法范式 |
|---|---|---|
| 分治 | O(V) 时间 / O(V) 空间 | 分治法（Divide and Conquer） |
| 贪心 / Kruskal MST | O(E α(V)) 时间 / O(V+E) 空间 | 贪心法（最小生成树） |
| 回溯 / DFS | O(V+E) 时间 / O(V) 空间 | 回溯法（深度优先搜索） |
| 分支限界 / BFS | O(E log E) 时间 / O(V+E) 空间 | 分支限界法（BFS + 优先队列） |

这四种算法的命名恰好对应了算法课中四类经典算法设计范式：分治、贪心、回溯、分支限界。它们分别代表了解题思路不同的思想流派——这也是为什么课程要求用这四种方法生成迷宫的原因。

## 附录：统计数据的生成方法

如果需要实际测量四种算法的统计数据，可以在项目中使用类似以下伪代码：

```cpp
for (auto algo : {分治, Kruskal, DFS, BFS分支限界}) {
    int sumSolution = 0, sumDeadEnds = 0, sumJunctions = 0, sumCorridor = 0;
    for (int seed = 0; seed < 100; ++seed) {  // 不同随机种子
        MazeModel maze;
        maze.generate(15, 15, algo, seed);
        auto stats = maze.statistics();
        sumSolution  += stats.solutionLength;
        sumDeadEnds += stats.deadEnds;
        sumJunctions += stats.junctions;
        sumCorridor  += stats.longestCorridor;
    }
    qDebug() << "算法" << algo
             << "平均解路径:" << sumSolution/100.0
             << "平均死胡同:" << sumDeadEnds/100.0
             << "平均交叉口:" << sumJunctions/100.0
             << "平均最长走廊:" << sumCorridor/100.0;
}
```
