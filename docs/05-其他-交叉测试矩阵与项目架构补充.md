## 1. 交叉测试矩阵的公平性问题

这是从课程评分规则中衍生出的一个非平凡问题。

### 1.1 课程规则原文

> 每组迷宫设计者提供一个"最佳迷宫"；将全部"最佳迷宫"组成迷宫基准测试集；每个 AI 玩家完成迷宫基准测试；每个迷宫的分数由全部 AI 玩家测评得到；每个 AI 玩家的分数由全部迷宫测评得到。

> AI 玩家分数定义：抵达终点时剩余资源价值 ÷ 所用步数。

### 1.2 核心矛盾

不同设计组会在迷宫内放置**不同数量的金币和陷阱**：

| 迷宫 | 金币×价值 | 陷阱×惩罚 | DP 最优得分上限 |
|---|---|---|---|
| 甲组 | 50×50 = 2500 | 10×(-30) = -300 | 2200 |
| 乙组 | 10×50 = 500 | 5×(-30) = -150 | 350 |

同一个 AI 玩家分别玩两个迷宫：

```
AI 在甲组迷宫上：剩余800资源，走了200步 → 得分 = 800/200 = 4.0
AI 在乙组迷宫上：剩余250资源，走了 80步 → 得分 = 250/80  = 3.125
```

**问题**：如果直接把所有迷宫上的 AI 得分取平均作为"AI 玩家总分"，那 AI 在甲组（资源多的迷宫）上的得分天然更高，甲组的迷宫就对 AI 的最终排名产生了不成比例的影响。同理，如果直接把各 AI 的得分平均作为"迷宫得分"，资源多的迷宫天然拿高分。

### 1.3 矩阵结构如何解决

交叉测试矩阵是：

```
           AI₁    AI₂    AI₃    AI₄     ← 每个AI在所有迷宫上打分
迷宫甲      s₁₁    s₁₂    s₁₃    s₁₄     
迷宫乙      s₂₁    s₂₂    s₂₃    s₂₄     
迷宫丙      s₃₁    s₃₂    s₃₃    s₃₄     
迷宫丁      s₄₁    s₄₂    s₄₃    s₄₄     
  ↑
每个迷宫被所有AI打分
```

$s_{ij}$ = AIⱼ 在迷宫ᵢ上的原始分数 = 剩余资源/步数。

公平性来自于一个简单但关键的事实：

> **同一行的分数是在同一个迷宫上产生的 → 金币/陷阱数相同 → 行内可比**
> **同一列的分数是同一个 AI 产生的 → 行之间绝对值不同，但排名消除差异**

具体来说：

**对迷宫评分（按行看）**：

迷宫甲上的四个分数 `[4.0, 3.2, 2.1, 1.5]` —— 虽然绝对值偏高（因为甲资源多），但这一行的**方差（区分度）**才是迷宫设计质量的体现。一个好迷宫应该让不同 AI 表现出明显差异（分数跨度大），而不是让所有 AI 拿分差不多。

**对 AI 评分（按列看）**：

AI₁ 的四个分数 `[4.0, 3.7, 4.5, 4.1]` —— 这是四个不同资源池的迷宫，绝对值不可直接与 AI₂ 的 `[3.2, 3.5, 3.8, 3.3]` 比较。但如果 AI₁ 在**每个迷宫上都排名第一**，那它就是最强的 AI。AI 评分应当以**排名聚合**（如平均排名）为准，而非原始分数的平均。

### 1.4 小结

| 比较维度 | 是否直接可比 | 正确做法 |
|---|---|---|
| 同一迷宫上 AI₁ vs AI₂ | ✅ 是 | 直接比分数（行内资源数相同） |
| 同一 AI 在迷宫甲 vs 迷宫乙 | ❌ 否 | 比排名（列内标准化） |
| 跨迷宫跨 AI 的原始分数 | ❌ 否 | 用矩阵排名消除资源差异 |

课程规则中说"每个迷宫的分数由全部 AI 玩家测评得到"，实际操作中合理的解释是：**对迷宫评分 = 该迷宫上所有 AI 得分的某种聚合（区分度/方差），而非原始平均**；**对 AI 评分 = 该 AI 在所有迷宫上的排名聚合，而非原始平均**。

### 1.5 项目内部的对应处理

项目在内部评估中（`MazeEvaluator`）通过**除以 DP 最优得分**做了显式归一化：

```cpp
aiResourceRatios[i] = clamp01(AI资源得分 / DP最优得分)
```

这使得不同资源数量的迷宫可以在比值层面公平比较（详见第 04 篇文档 2.1 节）。这不等同于课程交叉测试的评分公式，但体现了相同的设计思想。

### 2.2 线程模型

项目使用 Qt 的 `QThread` + `moveToThread` 模式：

```
主线程（GUI）                 工作线程① (GA)              工作线程② (AI)
────────────────              ──────────────              ──────────────
paintEvent()                 MazeOptimizer::run()        GreedyPlayer::play()
信号/槽分发                   for gen 0..N:              while !end:
QTimer 回调                    选择+交叉+变异+评估          选最优邻居→移动→收集
                              检查 stopped_              
stopOptimizer()                emit finished(best)       线程结束→finished回调
  → optimizer_->stop()         → 排队到主线程              → 排队到主线程
  → 断开所有信号               deleteLater() + quit()     deleteLater() + quit()
  → quit() + wait() + 清理
```

**关键设计**：
- GA 和 AI 都运行在独立线程，不阻塞 UI
- `std::atomic<bool> stopped_` 实现安全中断
- `generationId_` 计数器防止陈旧回调覆写新数据
- 生成新迷宫前调用 `stopOptimizer()` + `stopAiWorker()` 保证干净状态
- 线程栈设 8MB（默认可能不够 MazeOptimizer 的递归或 QVector 分配）

## 3. BOSS 战分支限界设计

### 3.1 核心算法

`BossSolver::solve()` 使用分支限界求解"最少回合击败所有 BOSS 的最优技能序列"。

```
状态空间：各 BOSS 血量 × 各技能冷却剩余回合数
分支策略：每层代表一个回合，选择一个可用技能对当前 BOSS 释放
剪枝：
  ① 当前回合数 + 乐观下界 ≥ 已知最优解 → 剪枝
  ② 同一状态已用更少或相同回合到达过 → 剪枝
  ③ 优先尝试高伤害技能 → 加速收紧上界
```

**乐观下界**：对每个技能独立计算"在剩余冷却状态下最多能用几次"，累加总伤害上限。如果总伤害上限仍不足以击杀剩余 BOSS，提升回合估计。该估计**允许同回合内不同技能的假想使用重叠**——因此它是乐观的（不会高估所需回合数），但比简单的 `ceil(剩余血量/最大伤害)` 更紧。

### 3.2 BOSS 战与 DP 资源的关系

```cpp
// BossSolver::solveWithMaze()
roundLimit = minTurns + 2;                          // 限定回合 = 最少回合 + 2
coinConsumption = max(1, dpMaxCoins - 1);           // 复活金币消耗
```

- AI 在迷宫中收集的金币用于 BOSS 战复活
- 如果金币不够复活 → GAME OVER
- 这建立了**迷宫资源收集与 BOSS 战**之间的耦合：资源收集不仅影响 AI 得分，还影响通关可能性

## 4. 动态规划资源收集

### 4.1 算法

完美迷宫是树，起点到终点主路径唯一。树形 DP：

```
f(u) = value(u) + Σ max(0, f(v))   （v 是 u 的不在主路径上的子节点）
```

- 金币 = +50，陷阱 = -30
- `max(0, f(v))`：子树收益 > 0 才进入，否则跳过
- 最终 walk 通过递归生成"深入分支 → 返回"的路径

### 4.2 复杂度

| 步骤 | 复杂度 |
|---|---|
| BFS 求主路径 | $O(V)$ |
| DFS 分支收益计算（带记忆化） | $O(V)$ |
| 选中分支的 walk 生成 | $O(V)$ |
| **总计** | **$O(V)$** |

## 5. JSON 导出格式

### 5.1 交叉测试格式（给 AI 组）

```json
{
  "maze": [["#","#","#"],["#","S","#"],["#","#","#"]],
  "B": [11, 13, 9, 15],
  "PlayerSkills": [[8,4], [2,0], [4,2], [6,3]],
  "minRouds": 20,
  "CoinConsumption": 5
}
```

字符含义：

| 字符 | 含义 |
|---|---|
| `#` | 墙壁 |
| ` ` (空格) | 普通通路 |
| `S` | 唯一起点 |
| `E` | 唯一终点 |
| `B` | BOSS 格（在通向 E 的必经之路上） |
| `G` | 金币（导出时转为 `C`） |
| `T` | 陷阱 |

**已知问题**：字段名 `minRouds` 是保留的拼写错误（应为 `minRounds`），因为课程接口要求保持一致。

### 5.2 GA 结果保存格式

  "format": "ga-result-v1",

## 7. 已知局限与改进方向

### 7.1 RL 玩家问题
| 问题 | 说明 |
|---|---|
| ε 衰减过慢 | 2000 episodes 后 ε≈0.18，远未到 0.03 |
| ε 每迷宫重置 | `trainOnMazes` 对每个迷宫重置 ε=0.5 |
| 距离奖励振荡 | 走近+0.01、走远不惩罚，agent 可靠振荡蹭奖励 |
| Q 表过大 | 4,194,304 条目 × 8 bytes ≈ 32MB |

### 7.2 适应度函数问题
| 问题 | 说明 |
|---|---|
| pathInefficiency 无上界 | AI 走满 maxSteps 时可达 ~2000 |
| 跨策略独立取最大 | 三个指标可能来自不同 AI 策略 |
| 传统公式与增强公式不一致 | 旧代码用 regret，新代码用 designGroupScore |

### 7.3 贪心 AI 局限
| 问题 | 说明 |
|---|---|
| 距离硬编码为 2 | ValuePerStep 退化为纯值排序 |
| NearestFirst 等价随机 | 所有邻居 score 相同，依赖遍历顺序 |
| AvoidTraps 兜底可走入陷阱 | 无资源可见时的兜底策略不过滤陷阱 |

### 7.4 协同进化问题
| 问题 | 说明 |
|---|---|
| bestFitness 初始为 0 | 负适应度永不记录 |
| 首轮 Q 表全零 | 第一次评估基于随机动作 |
