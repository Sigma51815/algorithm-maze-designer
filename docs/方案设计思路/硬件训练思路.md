# 硬件训练架构指南：15×V100 集群的强化学习工程实践

> **适用场景**：基于 15 张 V100 显卡 + 多核 CPU 服务器的训练环境  
> **核心目标**：最大化 GPU 利用率，支持多算法并发训练与超参搜索  
> **适用方向**：迷宫设计（遗传算法加固器）与 AI 玩家（DRQN/PPO 训练）

---

## 一、硬件负载特性与瓶颈分析

在传统深度学习中，GPU 是绝对主力；但在强化学习（RL）或程序内容生成（PCG）中，存在严重的**异构负载不均**现象。

### 1.1 CPU 瓶颈（数据收集/Rollout 阶段）

- 迷宫环境的逻辑（玩家移动、撞墙判定、扣血、吃金币、切出 3×3 局部视野）全部运行在 **CPU** 上
- 如果只开单进程环境，V100 在算完一次动作后必须等待 CPU 算完下一步迷宫状态
- **后果**：GPU 利用率极低（通常低于 5%）

### 1.2 GPU 爆发（反向传播/Backprop 阶段）

- 当 CPU 收集了足够的步数数据（Replay Buffer）并打包成 Batch 送入神经网络时，V100 的 Tensor Core 才会瞬间满载进行梯度更新
- **特征**：GPU 利用率呈脉冲式波动，大部分时间处于空闲状态

---

## 二、核心解决方案：向量化环境并行

为了解决 CPU 瓶颈，必须采用**异步多进程环境采样**架构。

### 2.1 工程实现

使用 Stable-Baselines3 的 `SubprocVecEnv` 或 Ray (RLlib) 框架。

### 2.2 工作原理

```
CPU 核心 1  ──→ 迷宫环境 1  ──→ 3×3 视野 ─┐
CPU 核心 2  ──→ 迷宫环境 2  ──→ 3×3 视野 ─┤
CPU 核心 3  ──→ 迷宫环境 3  ──→ 3×3 视野 ─┼──→ 打包成 Tensor ──→ V100
    ...                                    ─┤
CPU 核心 64 ──→ 迷宫环境 64 ──→ 3×3 视野 ─┘
```

- 在 CPU 上同时运行 64 个不同的迷宫环境进程
- 这 64 个环境同时执行 `step()` 并切出 64 个 3×3 的视野 Observation
- 将这 64 个输入打包成一个 Tensor，一次性喂给单张 V100 进行前向推理和反向传播

### 2.3 预期效果

完美解决 CPU 瓶颈，将 V100 的利用率拉满，训练速度提升数十倍。

## 四、Checkpoint 机制与交叉评测工作流

训练模型和最终提交/交叉 PK 的代码需要彻底**解耦**。

### 4.1 训练阶段：自动保存 Best Checkpoint

强化学习在探索新策略时模型可能产生性能波动，因此不能只保存最后的参数，必须使用评估回调函数（EvalCallback）。

**逻辑**：
1. 每隔 $N$ 步，代码自动在一个独立的验证集迷宫（不参与训练）中盲测 AI 的得分
2. 只有当 AI 刷新了历史最高得分记录时，才会覆盖保存为优化的权重文件（如 `best_model.zip`）

**伪代码**：

```
class EvalCallback:
    def __init__(self, eval_env, eval_freq, save_path):
        self.eval_env = eval_env
        self.eval_freq = eval_freq
        self.best_reward = -∞

    def on_step(self, step, model):
        if step % self.eval_freq == 0:
            reward ← evaluate(model, self.eval_env)
            if reward > self.best_reward:
                self.best_reward = reward
                model.save(save_path + "/best_model")
```

### 4.2 交叉评测阶段：纯前向传播模式

到了最终全矩阵交叉测试时，直接写一个干净的评测脚本，剥离所有训练逻辑。

**流程**：

```
1. 读取对方小组提供的未知迷宫矩阵（Excel/TXT/JSON）
2. 加载模型：model = load("./best_model.zip")
3. 前向推理循环：
   while not done:
       action = model.predict(obs, deterministic=True)
       obs, reward, done, info = env.step(action)
```

**关键注意点**：

必须设置 `deterministic=True`（确定性策略），关闭训练时为探索而存在的随机乱走机制，确保 AI 每一步都给出网络的绝对最优解。

## 五、选题后的冲刺方向

### 5.1 若选到 AI 玩家任务

**立即行动**：

1. 参考 `mazelab` 库，在 Python 中实现任务书要求的 3×3 视野切片和资源清算，封装为符合 Gymnasium 接口的 `Env` 类
2. 将环境部署到服务器上，挂载 Stable-Baselines3 框架
3. 用 V100 开始训练 DRQN/PPO 智能体

**技术栈**：

```
Python 3.8+
├── NumPy（迷宫矩阵操作）
├── Gymnasium（环境接口标准）
├── PyTorch（深度学习框架）
├── Stable-Baselines3（RL 算法库）
└── Ray（可选，用于分布式训练）
```

### 5.2 若选到迷宫设计任务

**立即行动**：

1. 用任务书要求的经典算法（分治/MST/DFS）作为迷宫生成的骨架
2. 在服务器上写一个遗传算法（GA）脚本
3. 使用遗憾值最大化作为适应度函数：

$$
\text{Fitness} = V_{\text{DP}} - V_{\text{Greedy}}
$$

4. 利用 V100 集群批量筛选能卡死通用 AI 的"地狱迷宫"

**技术栈**：

```
Python 3.8+
├── NumPy（迷宫矩阵操作）
├── 自实现四种迷宫生成算法
├── DP 最优路径算法
├── 3×3 贪心策略（用于计算 V_Greedy）
└── 遗传算法框架（自实现或使用 DEAP 库）
```

## 六、性能优化建议

### 6.1 环境优化

| 优化项 | 方法 | 预期效果 |
|:---:|:---|:---:|
| 环境加速 | 将迷宫逻辑用 C++ 实现，Python 调用 | 10-50x |
| 内存优化 | 使用共享内存存储迷宫矩阵 | 减少进程间拷贝 |
| IO 优化 | 使用内存映射文件存储 Replay Buffer | 减少磁盘 IO |

### 6.2 训练优化

| 优化项 | 方法 | 预期效果 |
|:---:|:---|:---:|
| 混合精度训练 | 使用 PyTorch AMP | 1.5-2x 加速 |
| 梯度累积 | 等效增大 Batch Size | 提升训练稳定性 |
| 学习率调度 | 使用 CosineAnnealing | 提升收敛速度 |
