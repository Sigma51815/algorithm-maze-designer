# 04 — 评估指标：优化效果的衡量体系

> 对应 goal.md 任务 (4)：理解评估指标如何确定优化效果——指标定义、计算方式、为什么能反映迷宫质量提升。

---

## 0. 核心问题

GA 优化迷宫的过程中，有一个根本问题需要回答：

> **"凭什么说优化后的迷宫比优化前好？"**

答案就是本项目精心设计的一套**多维度评估指标体系**。这套体系由三层组成：

```
第一层：原始观测数据
  └─→ 第二层：AI 表现指标（coinMissRate, trapHitRate, pathInefficiency）
        └─→ 第三层：设计质量指标（D / C / B → designGroupScore → finalFitness）
              └─→ 最终判定：优化前 vs 优化后对比
```

---

## 1. 第一层：原始观测数据

所有指标都源于对迷宫实际运行 AI 玩家后收集的原始数据。

### 1.1 数据采集方法

每次评估一个迷宫时，`MazeEvaluator::evaluate()` 执行以下操作：

```
Step 1: 保证资源已放置（或跳过，因为染色体已带资源）
Step 2: 运行 DP 最优解 → 获得 dpScore（理论上限）
Step 3: 统计迷宫中的总金币数 totalCoins 和总陷阱数 totalTraps
Step 4: BFS 计算最短路径长度 shortestPathLen（S→E 理想距离）
Step 5: 对四种贪心 AI 策略逐一运行：
          ValuePerStep, NearestFirst, AvoidTraps, EndGoalFirst
Step 6: 从每个 AI 的 PlayResult 中提取数据
```

### 1.2 PlayResult：每个 AI 运行一次的输出

```cpp
struct PlayResult {
    bool reachedEnd = false;       // 是否到达终点
    int totalSteps = 0;            // 总步数
    int remainingResource = 0;     // 剩余资源（初值 0，捡金币+50，踩陷阱-30）
    double score = 0.0;            // remainingResource / totalSteps（效率）
    int collectedCoins = 0;        // 收集的金币数
    int triggeredTraps = 0;       // 触发的陷阱次数
    QVector<int> walk;            // 完整行走路径
    bool bossDefeated = false;     // 是否击败 BOSS
    int bossAttempts = 0;         // BOSS 尝试次数
};
```

**关键数据项的物理含义**：

| 数据项 | 含义 | 备注 |
|---|---|---|
| `totalSteps` | AI 走了多少步 | 上限 = cellCount × 10，超时则终止 |
| `remainingResource` | AI 结束时持有的资源量 | 正值=净赚，负值=净亏 |
| `collectedCoins` | 实际捡到的金币数 | vs totalCoins |
| `triggeredTraps` | 实际踩到的陷阱次数 | vs totalTraps |
| `reachedEnd` | 是否通关 | 未通关的迷宫对部分玩家来说"太难" |

## 2. 第二层：AI 表现指标

### 2.1 每个指标的计算对象：最差 AI

评估时同时运行四种 AI 策略，然后**取表现最差的那个**作为该迷宫的 AI 表现：

```cpp
// maze_evaluator.cpp:82-120
for (GreedyStrategy s : strategies) {
    PlayResult r = GreedyPlayer::play(maze, {}, {}, 0, s);
    if (r.remainingResource < worst) {
        worst = r.remainingResource;
        // ← 最差资源得分的 AI 决定以下三个指标
        worstAIScore    = r.remainingResource / r.totalSteps;
        worstMissRate   = (totalCoins - r.collectedCoins) / totalCoins;
        worstHitRate    = r.triggeredTraps / totalTraps;
        worstInefficiency = (r.totalSteps - shortestPathLen) / shortestPathLen;
    }
    best = max(best, r.remainingResource);
}
```

**为什么取最差而非平均？** 如果取平均，迷宫设计者可以通过"让几种 AI 都表现中等"来获得中等分数。取最差强制要求迷宫对**所有**常见策略都有挑战性——只要有一种策略表现很好，这个迷宫就被认为不够难。

### 2.2 金币错失率（coinMissRate）

```
coinMissRate = (totalCoins - AI收集的金币数) / totalCoins
            = 1 - AI金币收集率
```

| 值 | 含义 |
|---|---|
| 0.0 | AI 捡到了所有金币（迷宫太简单） |
| 0.5 | AI 只捡到一半金币 |
| 1.0 | AI 一个金币都没捡到（但可能因为没有走到） |

**为什么这个指标重要？** 金币是迷宫中的"正向激励"。好迷宫应该把金币放在 AI 不容易到达的地方——迫使 AI 要么绕远路（增加步数），要么完全错过（降低得分）。高 missRate 说明资源位置对 AI 构成了有效挑战。

### 2.3 陷阱触发率（trapHitRate）

```
trapHitRate = AI触发的陷阱次数 / totalTraps
```

| 值 | 含义 |
|---|---|
| 0.0 | AI 完全避开了所有陷阱（AI 太聪明/陷阱位置太差） |
| 0.5 | AI 踩了一半陷阱 |
| 1.0 | AI 踩了所有陷阱（但不一定是坏事——DP 最优解可能也踩了） |

**注意**：trapHitRate 高不一定好。如果陷阱全部挤在主路径上导致所有 AI 都踩，DP 最优解也无法规避，那 regret 就小了。理想的陷阱位置是"AI 会踩但 DP 最优解可以规避"的位置——这正是**对抗模式把陷阱放交叉口**的原因：AI 反复经过交叉口时重复踩踏，DP 最优解一次性通过即可。

### 2.4 路径低效率（pathInefficiency）

```
pathInefficiency = (AI总步数 - 最短路径长度) / 最短路径长度
                 = (totalSteps / shortestPathLen) - 1
```

| 值 | 含义 |
|---|---|
| 0.0 | AI 走的就是最短路径（效率最高） |
| 0.5 | AI 多走了 50% 的步数 |
| 2.0 | AI 绕了三倍于最短路径的路 |

**这个指标捕捉的是"AI 绕了多少冤枉路"**。好迷宫应该有复杂的分支结构，诱使 AI 离开主路去探索分支（为了金币），然后在分支内走到底再回头——这就是 pathInefficiency 的来源。

### 2.5 三个指标的综合解读

一个"好迷宫"在三指标上期望的表现：

| 指标 | 理想方向 | 理想状态 |
|---|---|---|
| coinMissRate | ↑ 高 | AI 找不到大多数金币 → 金币藏得深 |
| trapHitRate | ↑ 高 | AI 踩到多数陷阱 → 陷阱放得阴险 |
| pathInefficiency | ↑ 高 | AI 绕了很多冤枉路 → 分支结构复杂 |

但三者需要**平衡**：如果 coinMissRate=1.0（一个金币没捡到），可能意味着 AI 根本到达不了金矿区，同时也可能 pathInefficiency 反而低（因为 AI 放弃了探索）。

## 3. 第三层：设计质量指标 D·C·B

第二层的三个指标描述的是"AI 多惨"，但**没有回答迷宫本身的设计质量**。比如一个把所有路都用陷阱堵死的迷宫，会让 AI 得分极低——但它不是一个"好迷宫"。

第三层 D·C·B 指标正是为了解决这个问题而设计的。

### 3.1 前置：AI 资源比值

在计算 D/C/B 之前，先将每种策略的绝对得分转化为相对比值：

```cpp
// 比值1: resourceRatio = AI 得分 / DP 最优得分（限制在 [0,1]）
aiResourceRatios[i] = clamp01(AI资源得分 / DP最优得分)

// 比值2: stepRatio = AI 得分 / AI 步数（效率值）
aiRatios[i] = remainingResource / totalSteps
```

然后计算四种策略比值的均值 `meanResourceRatio` 和标准差 `resourceStddev`。

### 3.2 D：区分度（Discrimination）—— 新公式

```
// 1. 每个AI的归一化得分 [0,1]
aiScore[i] = max(0, remainingResource[i] / totalSteps[i]);  // 每步效率
idealScore = (dpScore > 0 && shortestPathLen > 0)
    ? dpScore / shortestPathLen : 1.0;
normalizedScore[i] = clamp01(aiScore[i] / idealScore);
// 未到终点或步数为0 → 0

// 2. D = 标准差和极差的几何平均（双维度捕捉分散程度）
D = sqrt( clamp01(stddev(normalizedScores) / 0.22)
        * clamp01((max-min) / 0.55) )
```

**物理含义**：四种AI策略在迷宫上的归一化每步效率得分差异有多大。双维度设计同时捕捉：
- **标准差**：得分离散程度
- **极差**：最好和最差AI的得分差距

**为什么用每步效率而非绝对资源？** 绝对资源会受到迷宫总资源量的影响（资源越多，分数天然越高），无法公平比较。每步效率 `resource/steps` 是实际竞技指标——它不仅看"拿了多少"，还看"效率多高"。

**为什么用双维度（stddev × range）？** 单独使用stddev可能被"两个中等+两个极差"的分布骗过去；单独使用range可能被"一个异常值+三个扎堆"的分布骗过去。几何平均确保两者都有意义时D才高。

**极端情况分析**：

| 场景 | resourceStddev | D | 解读 |
|---|---|---|---|
| 四种策略得分完全一样 | 0.0 | 0.0 | 迷宫对不同策略无区分度 → 太简单/太随机 |
| 两种策略得 0.8，两种得 0.1 | ~0.35 | 1.0 | 完美：迷宫区分了"好策略"和"差策略" |
| 所有策略得分都很低但一致 | 0.0 | 0.0 | 迷宫太难但无区分 → 不是好设计 |

**D 的深层含义**：D 高意味着这个迷宫**不是一个"一视同仁"的死迷宫**——它能让聪明的策略占优，愚蠢的策略吃亏。这正是一个好的游戏关卡设计的特征。

### 3.3 C：稳定性（Stability）

```
C = clamp01( 到达终点的策略数 / 4.0 )
```

**物理含义**：在四种测试策略中，有多少比例能走到终点。

| C 值 | 策略到达数 | 解读 |
|---|---|---|
| 1.0 | 4/4 | 所有策略都能通关 → 迷宫是可完成的 |
| 0.75 | 3/4 | 一种策略卡住了 → 可接受 |
| 0.5 | 2/4 | 两种策略卡住了 → 偏难 |
| 0.25 | 1/4 | 三种策略卡住了 → 可能太极端 |
| 0.0 | 0/4 | 全部卡死 → 迷宫需要修复 |

**为什么 C 重要？** 如果没有任何 AI 能走到终点，D 和 B 的计算就没有意义——因为所有 AI 的得分都接近 0，比值没有参考价值。C 保证了"设计出来的迷宫至少对部分玩家是可通关的"。

### 3.4 B：难度适中（Balance）

```
B = clamp01( 1.0 - abs(nsMean - 0.45) / 0.30 )
```

**物理含义**：AI 的平均归一化得分是否接近"理想挑战水平"（0.45）。

**0.45 的来源**：AI 平均应拿到理论最大每步效率的 45%——既有挑战又不至于绝望。容忍区间 ±0.30（比旧版 ±0.18 更宽），给 D 优化留出更多空间。

### 3.5 设计组综合分 → 加权最终适应度

```
finalFitness = 100 × D^0.55 × B^0.30 × C^0.15
designGroupScore = finalFitness
```

**分数区间**：[0, 100]

**权重设计意图**：
- D（区分度）占 55%：核心目标——最大化不同AI得分差异
- B（难度适中）占 30%：约束——不能所有人失败或所有人满分
- C（可完成性）占 15%：底线——迷宫必须是可通关的

**为什么用指数加权而非几何平均？** 指数加权允许精确控制各维度贡献比例。D 是主角（55%），B 和 C 是约束条件。

### 3.6 finalFitness：GA 优化的实际目标

```cpp
result.finalFitness = result.designGroupScore;  // 直接等于设计组综合分
```

所以 GA 每代选择、交叉、变异的目标就是最大化 `designGroupScore`。

## 4. 补充指标

### 4.1 遗憾值（Regret）

```cpp
regretGreedy  = dpScore - worstGreedyScore    // DP vs 最差贪心
regretRL      = dpScore - rlScore             // DP vs RL 玩家
regretCombined = dpScore - min(worstGreedy, rlScore)
```

遗憾值是**传统适应度**的核心。它回答的问题是："最优解比最差 AI 多拿多少分？"

- regret 越大 → 迷宫对 AI 的"欺骗性"越强
- regret 接近 0 → 迷宫太简单，AI 几乎能拿到最优解

**但遗憾值有盲区**：如果迷宫中根本没放金币（totalCoins=0），dpScore=0，所有 AI 也拿 0 分 → regret=0 ——迷宫的"难度"是 0，但这显然不对。这就是为什么增强适应度不用 regret 而用 D/C/B 三维度——后者在资源为零时也会因为缺乏区分度和平衡性而给出低分。

### 4.2 拓扑难度（topoDifficulty）

详见第 03 篇文档 2.5 节。四个维度：
- 死胡同密度 ×2.0
- 最长走廊 /10 ×0.5
- 解路径迂回比 ×0.5
- 交叉口密度 ×1.0

最终 clamp 到 [0, 5]。

**topoDifficulty 与 AI 表现指标的互补性**：topoDifficulty 只依赖迷宫拓扑结构，不依赖资源分布。它的存在保证了即使资源模式改变（如从对抗改为随机），迷宫的"骨架难度"仍能被评估。

### 4.3 RL 评分（可选）

如果 `evaluateAgainstRL = true`，额外运行 RL 玩家评估，得到：
- `rlScore`：RL 玩家的资源得分
- `regretRL`：DP - RL 的遗憾值

RL 评分主要用于**协同进化**（CoEvolution）场景。

## 5. 优化前 vs 优化后的对比框架

### 5.1 对比触发时机

每次 GA 优化完成后，UI 自动生成对比报告（`mainwindow.cpp:1346-1391`）：

```
GA 完成
  │
  ├─ 对优化前迷宫（preOptMaze_）运行 MazeEvaluator::evaluate() → before
  ├─ 对优化后迷宫（optimizedMaze_）运行 MazeEvaluator::evaluate() → after
  │
  └─ 生成 HTML 对比表
```

### 5.2 对比表的 7 个指标

| 对比项 | 含义 | 好趋势 |
|---|---|---|
| 金币漏拾率 | AI 错过金币的比例 | ↑ |
| 陷阱命中率 | AI 踩中陷阱的比例 | ↑ |
| 路径迂回度 | AI 绕路程度 | ↑ |
| 区分度 D | 不同 AI 策略表现差异 | ↑ |
| 稳定性 C | AI 到达终点的比例 | 保持（不要太低） |
| 均衡性 B | AI 表现接近目标水平 0.45 | ↑ |
| **迷宫组代理分** | **综合设计质量分** | **↑** |

### 5.3 判定逻辑

```cpp
bool improved = after.finalFitness > before.finalFitness;
if (improved) {
    // ✅ "优化有效（迷宫组代理分提升）"
} else {
    // ❌ "优化未提升迷宫组代理分"
}
```

**判定标准非常简单**：优化后的 `designGroupScore` > 优化前的 → 优化有效。

### 5.4 对比表实际 UI 样式

```
对比评估（优化前 vs 优化后）
┌──────────────────┬──────────┬──────────┐
│                  │  优化前   │  优化后   │
├──────────────────┼──────────┼──────────┤
│ 金币漏拾率        │  45.2%   │  67.8%   │
│ 陷阱命中率        │  32.1%   │  58.4%   │
│ 路径迂回度        │  125.3%  │  198.7%  │
│ 区分度 D          │  0.42    │  0.78    │
│ 稳定性 C          │  0.75    │  0.75    │
│ 均衡性 B          │  0.56    │  0.82    │
│ 迷宫组代理分       │  72.3    │  88.5 ✓  │
└──────────────────┴──────────┴──────────┘

✓ 优化有效（迷宫组代理分提升）
```

## 7. 指标设计的核心思想

### 7.1 多维度 > 单维度

如果只用 "AI得分" 一个指标，GA 会趋向于产生让 AI 完全走不到的极端迷宫（高 coinMissRate、高 trapHitRate，但 C=0）。D/C/B 三维度强制了设计质量的最低门槛。

### 7.2 对抗性评估

评估的 AI 不是"最优 AI"，而是**四种不同策略的贪心 AI**。这模拟了真实场景中不同水平的玩家——设计者不能只为一种玩家类型优化。

### 7.3 DP 最优解作为基线

所有 AI 表现都相对于 DP 最优解来衡量。这提供了公平的比较基础——不同迷宫有不同数量的金币/陷阱，DP 最优解得分为这些差异提供了归一化。

### 7.4 分层指标体系

```
原始数据层         AI 表现层          设计质量层          最终判定
┌─────────┐     ┌───────────┐     ┌─────────────┐     ┌────────┐
│totalSteps│     │missRate   │     │             │     │        │
│coins     │ ──▶ │hitRate    │ ──▶ │ D · C · B   │ ──▶ │ 判定   │
│traps     │     │inefficiency│    │    ↓        │     │ 有效？ │
│reachedEnd│     │           │     │groupScore   │     │        │
└─────────┘     └───────────┘     └─────────────┘     └────────┘
```

每一层都建立在前一层的基础上，逐步从"AI 表现如何"过渡到"迷宫设计是否好"。

## 8. 已知局限与注意事项

### 8.1 设计组代理分的区间限制

- 最小值 60，最大值 100，区间仅 40 分——对于"极端差"和"优秀"的区分度可能不够
- 当 D/C/B 中有任何一个接近 0 时，几何平均急剧下降 → 这种情况是设计意图

### 8.2 AI 策略的局限

- 只测试了 4 种固定贪心策略，未覆盖所有玩家类型（如随机漫步、RL 策略等）
- evaluateAgainstRL 是可选开关，默认关闭（因为训练 RL 耗时）
- GreedyPlayer 的 maxSteps = cellCount × 10：对于 15×15 迷宫是 2250 步，对某些极端迷宫可能不够

### 8.3 资源分布的评估耦合

- 优化前和优化后的资源分布可能完全不同（GA 会重新放置资源），所以对比不单纯是"拓扑优化"
- 如果资源数量在优化前后不同（不应该发生但可能因为 autoCoinCount 不一致），对比会失真

### 8.4 B=0.45 目标的主观性

0.45 是设计者选择的"理想挑战水平"。不同类型的游戏可能需要调整这个参数。目前是硬编码的。

### 8.5 topoWeight 参数的作用

虽然 `topoWeight = 0.3` 存在，但在增强适应度中 `finalFitness = designGroupScore`，topoDifficulty 只在 `computeTopoDifficulty()` 被调用时计算，并没有直接乘入 fitness。它更多是一个备用/参考指标。

| 参数 | 值 | 用途 |
|---|---|---|
| D 归一化分母(stddev) | 0.22 | nsStddev / 0.22 → D的stddev分量 |
| D 归一化分母(range) | 0.55 | nsRange / 0.55 → D的range分量 |
| D 公式 | sqrt(clamp01(stddev/0.22) × clamp01(range/0.55)) | 双维度区分度 |
| B 目标均值 | 0.45 | AI 平均应达到理想效率的 45% |
| B 容忍范围 | 0.30 | ±0.30 内 B=1.0，超出线性下降 |
| C 公式 | reachedCount / 4 | 到达终点的策略比例 |
| finalFitness | 100 × D^0.55 × B^0.30 × C^0.15 | [0, 100] 加权适应度 |
| topoDifficulty 区间 | [0, 5] | clamp 确保不溢出 |
| GreedyPlayer maxSteps | cellCount×10 | 15×15 → 2250 步 |
| AI 测试策略数 | 4 | ValuePerStep, NearestFirst, AvoidTraps, EndGoalFirst |
| 未到终点处理 | 得分记为 0 | 防止"全员失败=好迷宫" |
