# 任务②③算法思路说明

本文档用于配合代码批注阅读。代码里只保留函数意义和关键变量变化，较完整的讲解放在这里。

## 任务②：动态规划资源收集

对应代码：

- `src/source/core/maze.cpp`：`MazeModel::optimalResourceWalk()`
- `src/source/ui/mazewidget.cpp`：DP 路径和分支高亮绘制
- `src/source/ui/mainwindow.cpp`：点击“DP 求最优路径”后的结果展示

### 1. 这部分代码的意义

任务②要求求出迷宫中能获得的最大资源数，并且给出一条合法路径。资源规则是：金币加分，陷阱扣分，同一个格子的金币或陷阱只在第一次经过时计分。

本项目生成的是完美迷宫，也就是任意两个格子之间只有唯一通路。因此可以把迷宫通路看成一棵树。树上没有环，一个分支内部能贡献多少资源，不会受到另一个分支影响，所以可以用树形 DP 计算每个分支的净收益。

### 2. 核心变量含义

`calculateGain(cell, from)`：
表示从 `from` 走进 `cell` 这一侧子树后，这一侧最多能贡献多少净资源。`cell` 是当前格子，`from` 是来时的格子，用来避免递归走回头路。

`directedGain[cell][from]`：
缓存 `calculateGain(cell, from)` 的结果。因为代码会尝试不同根节点，相同方向的分支收益可能被重复查询，缓存可以避免重复递归。

`bestRoot` 和 `bestValue`：
代码会枚举每个格子作为收集区域的根节点，选出净收益最大的那个根。`bestRoot` 是最优区域的起点，`bestValue` 是当时算出的最大净收益。

`selected`：
保存 DP 最终决定要进入的格子集合。它从 `bestRoot` 开始，只向 `calculateGain(next, cell) > 0` 的子分支扩展。

`branchDecisions`：
保存每个分支的取舍记录，包括分支入口、净收益、是否被选择。界面上蓝色表示选中分支，红框表示放弃分支。

`walk`：
实际要画出来的行走路径。代码用 DFS 方式走进分支、收集、再返回，所以 `walk` 里允许出现回访。

`collectedCells`：
记录第一次经过的格子。最终资源只按这个集合计分，避免回访时重复加金币或重复扣陷阱。

### 3. DP 转移怎么理解

代码中的核心转移是：

```cpp
value += std::max(0, calculateGain(next, cell));
```

含义是：当前分支如果净收益为正，就值得进入；如果净收益小于等于 0，就不进入，把它当作贡献 0。

这就是“为何要权衡拿不拿某些金币”。有些金币可能在分支深处，但为了拿它需要经过陷阱。如果整条分支算下来净收益不是正数，DP 就会放弃它。

### 4. 路径为什么合法

`walk` 只通过 `neighbors(cell)` 添加下一格。`neighbors()` 返回的是当前格真实连通的相邻格，所以路径不会穿墙，每一步都是上下左右相邻。

DFS 回放时会出现“走进分支后再走回来”的情况，这是合法回访。资源计分时用 `collectedCells` 去重，所以金币和陷阱只在第一次经过时生效。

### 5. 验收时可以这样讲

“因为完美迷宫是一棵树，所以我用树形 DP 算每个分支的净收益。`calculateGain(cell, from)` 表示从 `from` 进入 `cell` 这一侧后最多能拿多少资源。转移时只累加正收益分支，非正收益分支不走，这样就能权衡拿不拿某些金币。最后把选中的格子 DFS 回放成实际路径，路径每一步都来自相邻通路；资源只在首次经过时计分。”

## 任务③：分支限界 BOSS 战

对应代码：

- `src/source/game/bosssolver.cpp`：`BossSolver::solve()`、`optimisticTurnLowerBound()`、`verify()`
- `src/source/ui/mainwindow.cpp`：求解并输出最优序列

### 1. 这部分代码的意义

任务③要求在技能冷却约束下，用最少回合击败所有 BOSS，并输出可验证的技能序列。

分支限界的思路是：搜索所有可能的技能释放序列，但不是盲目展开。先用一个可行解得到上界，再用乐观下界判断某些分支不可能更优，就提前剪掉。

### 2. 状态和分支

一个战斗状态由两部分组成：

- `health`：每个 BOSS 的剩余血量。
- `cooldowns`：每个技能还剩多少回合冷却。

每一层搜索代表一回合。当前回合可以选择的技能，就是 `cooldowns[i] == 0` 的技能。每个可用技能形成一个搜索分支。

`applySkill()` 会完成一回合仿真：扣 BOSS 血量、让所有技能冷却减少、把刚使用的技能重新设为冷却。

### 3. 上界怎么来

代码先用一个简单贪心策略打一遍：每回合选择当前可用技能中伤害最高的技能。这个贪心序列不一定最优，但它是一个合法解。

这个合法解的长度就是初始上界 `bestTurns`。后续搜索只需要找比它更短的序列。如果一个分支已经不可能更短，就可以剪枝。

### 4. 下界怎么剪枝

下界由 `optimisticTurnLowerBound()` 计算。最基本的下界是：

```text
ceil(剩余总血量 / 单回合最大伤害)
```

例如剩余 31 血，最大技能伤害是 10，那么至少要 4 回合。

代码还进一步考虑冷却：高伤害技能可能不能每回合都用，所以会估算未来若干回合内，在当前冷却状态下最多能打出多少伤害。如果这个最大伤害容量都打不死剩余血量，说明回合数不够。

如果：

```text
当前已用回合 depth + 乐观下界 optimisticTurns >= 当前上界 bestTurns
```

说明这个分支即使按最理想情况也不可能比当前答案更短，就可以剪掉。

### 5. 状态记忆怎么剪枝

`stateKey(health, cooldowns)` 把当前 BOSS 血量和技能冷却拼成一个状态键。

如果同一个状态以前已经用更少或相同回合到达过，那么当前分支没有继续搜索的必要。因为从相同状态出发，后续可选技能完全一样，而当前已经不占优势。

### 6. 技能序列如何验证

`BossSolver::verify()` 不做搜索，只按输出的技能序列逐回合重放。

每回合检查：

- 是否还有 BOSS 没被击败；
- 技能编号是否合法；
- 技能冷却是否为 0；
- 使用技能后是否最终击败所有 BOSS。

只有全部通过，界面才输出结果。

### 7. 验收时可以这样讲

“BOSS 战用分支限界搜索最短技能序列。每个状态由 BOSS 剩余血量和技能冷却组成。先用贪心得到一个合法技能序列作为上界；再用剩余血量除以最大伤害得到乐观下界，并结合冷却估算最大伤害容量。如果当前回合数加下界已经不可能优于上界，就剪枝。同时用状态记忆避免重复搜索。最后用 `verify()` 逐回合仿真，确认技能冷却合法、BOSS 按顺序被击败、序列长度等于最少回合数。”
