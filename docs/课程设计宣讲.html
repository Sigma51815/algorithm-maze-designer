🎮这门课在做什么

设计一个由经典算法驱动的迷宫探险游戏。全班分成两类组,一类造迷宫,一类闯迷宫,最后让两类组互相测试、互相打分。

🏗

迷宫设计组

生成一个探险流程完整、资源与关卡有挑战的迷宫:金币、陷阱、守终点的 BOSS。

产出:迷宫 + 最优资源解 + BOSS 战参数

🤖

AI 玩家组

写一个 AI 玩家:从起点出发,捡金币、避陷阱、击败 BOSS,最终抵达终点。

产出:自动闯关程序,追求资源多、步数少

🔗 两类组互为对手:设计组造的迷宫要给 AI 组去闯;AI 组闯得好不好,反过来评价迷宫质量。你的成果会成为别组的考题,别组的成果也会来检验你——这就是第 8 次课的"交叉验证"。

👥第一步:分组与抽签

  • 3~4 人一组,每组设一名组长(负责协调、提交、签字确认分工)
  • 抽签决定你们组是 迷宫设计组 还是 AI 玩家组(不可自选)
  • 注册登录系统平台,在左侧"必做题"区查看你们的任务(每类组各 4 道)
  • 报告需标注每位组员负责的部分——这关系到个人分
💡 建议组内做好分工:有人主攻算法、有人做可视化、有人写报告,但每个人都要能讲清本组的核心算法。

📌 成绩两步走

  • 先按小组给一个组分(满分 100)
  • 再按组内贡献比例折算到每个人
  • 最终成绩为五分制(优/良/中/合格/不合格)
分工要写清、贡献要真实——"搭便车"会在个人分上明显体现。

🧮这门课的考点:五种算法范式

每个任务都绑定一种算法范式。验收时不仅看结果对不对,更看你是否真用了指定的算法,并能讲清它。

范式用在哪一句话核心
分治迷宫生成大问题切成小问题,分别解决
贪心迷宫生成(MST)、3×3 拾取每步都选当前最划算的
回溯迷宫生成(DFS)一条路走到黑,不通就退回换路
动态规划资源收集存下子问题答案,组合出大问题答案
分支限界迷宫生成(BFS)、BOSS 战搜索时用"界限"剪掉没希望的分支
⚠ 验收会让你现场运行 + 口头讲解算法,还可能临时改输入让你重跑。只把结果抄对、却讲不清算法,或偷换成别的方法,都拿不到分。

🗺统一的迷宫格式(地基)

两类组都用同一套迷宫 JSON 格式,这样彼此的文件才能互通、交叉验证才跑得起来。详见 input说明.txt。

字符含义价值
#墙(不可走)—
空格通路0
S / E起点 / 终点0
B守终点的 BOSS—
G金币+50
T陷阱−30
{
  "maze": [ ["#","S",...], ... ],
  "B": [11, 13, 9, 15],
  "PlayerSkills": [[8,4],[2,0]],
  "minRouds": 20,
  "CoinConsumption": 5
}
  • 除 # 外都可走;上下左右相邻即连通(不算斜向)
  • S/E 开在最外圈边界作入口/出口
  • G/T 只在首次经过时计分
⚠ 字段名是 minRouds(不是 minRounds),按样例既定拼写,别擅自改,否则两组文件读不到一起。

🏗迷宫设计组:三个任务

①

多方法生成迷宫

用四种算法各生成一个迷宫,并对比复杂度与风格。

分治贪心回溯分支限界

交:4 个迷宫 + 对比分析

②

DP 资源收集

用动态规划算出迷宫里能拿到的最多资源及路径。

动态规划

交:最大资源数 + 路径 + 可视化

③

分支限界 BOSS 战

用分支限界算打 BOSS 的最少回合数与最优技能序列。

分支限界

交:最少回合 + 技能序列

这三项分布在不同次课验收;最后还要把成果整合成一个"最佳迷宫"提交,供 AI 组在交叉验证里闯。接下来三页逐个讲清每个任务要交什么。

①多方法迷宫生成

输入迷宫尺寸 n,用分治、贪心、回溯、分支限界四种算法各生成一个 n×n 迷宫。

再从时间复杂度、空间复杂度、挑战性三个角度对比分析。

✅ 必须满足

  • 连通:无孤立区域,处处可达
  • 唯一通路:任两点只有一条路("完美迷宫")
  • 尺寸够大、能放下较多资源(如 15×15+)
  • 能可视化展示生成过程(不是只给最终图)

②动态规划资源收集

给定一个迷宫(金币 +50,陷阱 −30),起点可任选,求能拿到的最多资源及对应路径。

  • 沿上下左右四方向移动,不算斜向
  • 同一格的 G/T 仅首次经过时计分
  • 输入沿用统一迷宫 JSON 格式

📤 要交什么

  • 最大资源数 主要判定指标
  • 对应的收集路径
  • 路径长度
  • 可视化结果(路径叠在迷宫上)
资源数会和标准答案对照,必须一致;路径合法即可,长短不强求。

③分支限界 BOSS 战

输入:一组 BOSS 血量 + 玩家技能(每个技能含伤害和冷却)。按顺序击败所有 BOSS。

  • 每回合放一个当前可用技能
  • 技能放出后要冷却几回合才能再用
  • 伤害溢出:打死当前 BOSS,多余伤害顺延到下一个
🧮 例:BOSS [5,10],这回合打 8 → 5 血的死、剩 3 伤害溢出,下一个 BOSS 变 7 血。

📤 要交什么

  • 最少回合数 硬指标
  • 最优技能序列(可仿真验证)
  • 给 AI 玩家的:限定回合数、复活金币消耗
回合数比标准答案更少且可验证= 更优,不算错。

🤖AI 玩家组:两个任务

①

局部贪心拾取

在 3×3 小视野里用贪心策略捡资源,让"资源数 ÷ 步数"尽量高。

贪心

考查:能否设计好的局部决策

②

完整迷宫探险

在整张迷宫里:捡金币、避陷阱、打 BOSS、抵达终点。

综合

考查:端到端的通关效率

🚨 贪心任务的命门:AI 只能看见周围 3×3,不能"开天眼"看全图——不准用 BFS/DFS/DP/穷举找全局最优。验收会查代码,长期等于全局最优解的会被重点检查。

①局部贪心资源拾取

玩家 P 在 3×3 正中央,周围随机分布金币 G、陷阱 T、空地 .。每步选"最划算"的方向走。

{ "grid": [
   [".","T","G"],
   [".","P","T"],
   [".","T","G"] ] }

📤 要交什么

  • 拾取资源数
  • 路径步数
  • 资源数 ÷ 步数 比值 越大越好
  • 实时拾取过程可视化
验收要当面讲清你的贪心评价函数,并确认只用了 3×3 局部信息。

②完整迷宫探险

从 S 出发,完成捡资源 + 打 BOSS + 到 E 的全过程。

  • BOSS 战有限定回合,打不过要消耗金币复活
  • 复活后已击败的 BOSS 不回血
  • 既要会捡资源,也要会打 BOSS、算好金币够不够复活

📤 要交什么

  • 剩余资源数
  • 总步数
  • 剩余资源 ÷ 总步数 核心分数
  • 完整过程可视化
📊 这个"剩余资源 ÷ 总步数"比值就是交叉验证矩阵里要填的那个数——越高越好,直接决定你在所有 AI 组里的排名。

🔗交叉验证:两类组互相打分

1
每个设计组提交一个"最佳迷宫"
2
每个 AI 组在所有最佳迷宫上跑一遍
3
记录每个迷宫上的"剩余资源 ÷ 步数"
4
形成"迷宫 × AI"矩阵,算出两类组得分

怎么算分

  • AI 组:在各迷宫上"相对其他 AI"的表现求平均
  • 设计组:迷宫能否拉开 AI 差距 + 难度是否适中
,AI1,AI2,AI3
迷宫A,1.2,0.8,1.0
迷宫B,2.1,1.5,1.8
好迷宫不是最难,而是难度适中、能分出高下;太简单或太难都拿不到高分。

📊成绩怎么来 · 报告要求

成绩构成

  • 代码测评:平台测评 + 代码质量
  • 阶段测试:现场运行 + 口头讲解
  • 交叉验证:互测得分
  • 课题报告:质量与完整度
四项合成小组分 → 按组内贡献折算到个人 → 全班排序、按学院比例定五分制等级(优/良/中/合格/不合格)。

📄 报告硬要求

  • 《计算机学报》模板,≥ 7 页
  • 每个算法的伪代码/流程图
  • 每个算法的复杂度分析(含过程)
  • 图/表展示实验数据
  • 算法的创新点
  • 标注各人撰写部分(关系个人分)
