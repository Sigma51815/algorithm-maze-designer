## 一、四种基础算法生成迷宫

### 1.1 功能验证

| 检查项 | 操作 | 预期结果 |
|--------|------|---------|
| 分治 (Divide & Conquer) | 算法选「分治」→ 点「生成并可视化迷宫」 | 迷宫完美生成，验证标签绿色，显示 `合法完美迷宫` |
| Kruskal MST | 算法选「贪心 / Kruskal MST」→ 同上 | 同上 |
| DFS | 算法选「回溯 / DFS」→ 同上 | 同上 |
| BFS | 算法选「分支限界 / BFS」→ 同上 | 同上 |

### 1.2 连通性验证

- 点「生成」后，验证标签必须显示 `合法完美迷宫：V=N, E=N-1，连通且 E=V-1`
- 如果变红，说明算法产生的迷宫有孤立区域或环 — **必须修复**
- 用不同 seed（种子框输入随机值）重复 3 次，确认每次都通过

### 1.3 动画验证

- 速度调到 50ms，点「生成」— 应该看到墙壁逐条出现
- 速度调到 1ms，点「生成」— 应该瞬间完成，迷宫完整显示
- S（起点）、B（BOSS）、E（终点）标记位置正确

### 1.4 尺寸变化

| 行×列 | 预期行为 |
|-------|---------|
| 5×5 | 正常生成，验证通过 |
| 15×15 | 正常生成，资源约 27 金币 / 18 陷阱 |
| 31×31 | 正常生成，资源 capped 到 50 / 35 |

## 二、基础算法 + 优化开关生成迷宫

### 2.1 优化流程

| 步骤 | 操作 | 预期 |
|------|------|------|
| 1 | 选算法（如「分治」），勾选「启用遗传算法优化」 | 右侧优化参数解锁 |
| 2 | 保持默认参数（种群16/代数30/变异率15%/对抗模式ON） | — |
| 3 | 点「生成并可视化迷宫」 | 先快速生成基础迷宫，然后 GA 开始迭代 |
| 4 | 等待优化完成 | 进度条逐代更新；完成后**自动显示对比表** |
| 5 | 检查对比表 | 显示「优化前 vs 优化后」的三维指标和综合难度分 |

### 2.2 算法一致性

**关键检查**：优化必须基于当前选中的基础算法，不可跨算法。

| 操作 | 验证方法 |
|------|---------|
| 选「分治」→ 优化 | 对比表的「优化前」应来自分治算法生成的迷宫 |
| 选「DFS」→ 优化 | 同上，优化前应为 DFS 迷宫 |
| 切换算法后再次优化 | 每次对比应独立，不应出现其他算法的迷宫 |

### 2.3 对抗模式 toggle

| 对抗模式 | 预期 |
|---------|------|
| ON（默认） | 陷阱放分叉口，金币藏分支深处。AI 容易踩陷阱、难找到金币 |
| OFF | 智能放置：金币放死胡同、陷阱放分叉口。对比两种模式后的难度分**应可比**（使用同一复合公式） |

## 三、优化 vs 不优化的对比指标自洽性

### 3.1 核心原则

> **优化后的迷宫难度必须 ≥ 优化前的迷宫难度。**  
> 如果出现「优化后 AI 得分反而更高」或「综合难度分下降」，说明适应度函数或对比逻辑有 bug。

### 3.2 检查方法

| 步骤 | 操作 |
|------|------|
| 1 | 不勾优化，用分治生成迷宫 → 跑 AI（「运行贪心 AI」） → 记录 `remainingResource` 和步数 |
| 2 | 勾优化，同样分治 → 点「生成并可视化迷宫」→ 等待完成 |
| 3 | 自动显示的对比表中，检查**综合难度分**：`优化后 > 优化前` 必须成立 |
| 4 | 如果出现 `优化后 ≤ 优化前`，检查三个子指标：
|    | - `金币漏拾率`（优化后应 >= 优化前） |
|    | - `陷阱命中率`（优化后应 >= 优化前） |
|    | - `路径迂回度`（优化后应 >= 优化前） |
|    | - 如果三个子指标有下降，说明 GA 进化方向有问题 |
| 5 | 换 DFS/BFS/Kruskal 各重复一次 |

### 3.3 适应度公式自查

适应度 = `coinMissRate × 50 + trapHitRate × 30 + pathInefficiency × 20 + topoWeight × topoDifficulty × 50`

- **coinMissRate**: AI 漏捡的金币比例 (0~1)，越高越好
- **trapHitRate**: AI 踩中陷阱的比例 (0~1)，越高越好
- **pathInefficiency**: AI 多走的步数/最短路径 (0~N)，越高越好
- **topoDifficulty**: 拓扑难度因子，由死胡同密度、分叉口密度、走廊长度等计算

**验证**: 取一个已知难度的迷宫，手动跑 4 种贪心策略，计算以上指标，对比 GA 输出的 fitness 是否一致。

### 3.4 对比评估界面检查

- 对比表标题应显示「优化前 vs 优化后」（不是「原迷宫 vs 优化后」— 避免自己比自己）
- 对比表应包含 4 行指标 + 1 行结论（`✓ 优化有效（难度分提升）` 或 `✗ 优化未变难`）
- 「保存最佳迷宫」按钮可点击，弹出保存对话框，文件可写入

## 四、程序崩溃测试

### 4.1 快速连续点击测试

> **这是最常见的崩溃场景。** 以下每个测试重复 3 次。

| 场景 | 操作 | 预期 |
|------|------|------|
| 快速点击「生成」 | 连续点 5 次，间隔 < 200ms | 不崩溃，不闪退 |
| 优化中点击「生成」 | 先点优化，在 GA 运行中再点「生成」 | 不崩溃；新生成应停止旧优化 |
| 优化中反复 toggle 优化开关 | GA 运行中，勾/不勾优化 checkbox | 不崩溃 |
| 快速切换算法 | 反复选择不同算法 | 复杂度描述切换，不崩溃 |
| 优化完成后快速点对比/保存/AI | 连续操作 | 不崩溃 |
| AI 运行中生成新迷宫 | AI 动画中点击「生成」 | 不崩溃；AI 应被停止 |
| AI 运行中应用优化 | AI 动画中点击（如果有） | 不崩溃 |

### 4.2 边界条件

| 场景 | 操作 | 预期 |
|------|------|------|
| 先优化再生成再优化 | 完整来回 | 不崩溃 |
| 金币/陷阱设为 0 | 生成迷宫 | 正常，无资源 |
| 金币/陷阱设为 300（最大） | 生成 5×5 迷宫 | 自动 clamp，不崩溃 |
| 种子设为边界值（1 或 999999999） | 生成迷宫 | 正常 |

### 4.3 线程安全

- GA 优化在 QThread worker 中运行，主线程不应卡死
- AI 运行在 QThread worker 中运行，主线程不应卡死
- `stopAiWorker()` 和 `stopOptimizer()` 在析构函数中被调用，退出程序时不应 crash
- 关闭窗口时如果有正在运行的优化/AI，应先停止线程再退出

## 七、已知可接受问题

1. **RL 自测 flaky**: `FAIL RL score decreased` 在某些 run 中会出现。这是因为 Q-Learning 用 2000 episode 无法保证单调提升。与 GA/优化逻辑无关。重跑即可通过。
2. **`minRouds` JSON key 拼写**: 应为 `minRounds`，但为保持交叉测试兼容性不改。
3. **MacBook Air 无散热**: 不建议长时间运行优化（代数 > 50 或种群 > 30），CPU 可能过热降频。
